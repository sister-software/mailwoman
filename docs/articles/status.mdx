# Status

This page is the single source of truth for what works in Mailwoman right now.

:::info[Verified as of 2026-06-25 — release 4.15.0]
The package version + model lineage below are from the shipped `model-card.json` (v4.15.0 =
`v1.9.3a3-anchor-absorption`). v4.15.0's headline is the **#723 postcode-anchor fix**: a leading
5-digit token that is actually a US house number which happens to look like a ZIP (`12345 Main St`)
is now correctly labeled `house_number` with the postcode anchor ON — the `SLICE-H` case went
**20 → 100**, at **zero coordinate cost** (non-circular oa-resolver-eval, 2000 real OA-US points:
whole-US locality-match 97.7 → 97.8, region 99.9 flat, coord p50 3.4 km flat). It trades
us.postcode label-F1 −2 — a rare leading-postcode (VT E911) case that is coordinate-invisible (VT
locality flat 92.6 = 92.6); the gate floor was revised with an operator-approved, stated reason. The
per-tag **F1 tables below are still from the v4.4.0 ship gate** (2026-06-11) — the last full per-tag
scorecard; treat them as the v4.4.0 baseline, not v4.15.0. The
[parity scorecard](./evals/parity-scorecard-2026-06-11.md) is the live per-tag truth; for which
release added which capability, see [Releases & capabilities](./releases.mdx).
:::

The trend, in words:

- **Healthy:** postcode (98/99 US/FR), house_number (96/91), venue-US (90), street (79 US).
- **Fixed by the parity campaign:** `unit` 0 → 92.3 on real-OOD designators (the 4.1.0 headline).
- **Shipped by the consolidation (v4.2.0):** locality +12.8, region +10.7, country 0 → 89.8
  (gazetteer soft anchor), affixes now exist (0 → 64.9/48.8 at P≈100), DE 90.9.
- **Closed by the boundary consolidation (v4.4.0):** the scorecard's last empty rows — po_box
  0 → 89.1, cedex 0 → 96.1, intersections 0 → 100 (real-OOD, P ≥ 95) — and the v4.3.0 arena
  debt (perturb 64 → 72, now a gated floor). The span bridge ships as required decode config:
  dotted surfaces ("P.O. Box") are structurally unlabelable in the corpus format; the bridge is
  the containment ([ship gate](./evals/2026-06-11-v4.4.0-ship-gate.md) tells the three-battery story).
- **Closed by the relabel (v4.3.0):** the affix ceiling was never capacity — it was a 1,039:1
  label contradiction in the training mix ([#492](https://github.com/sister-software/mailwoman/issues/492) →
  [#511](https://github.com/sister-software/mailwoman/issues/511)). Affixes now 93.6/96.6 (int8,
  real-OOD; 92.2/90.3 on the 193-row NAD-native eval where v4.2.0 scored 18.2/8.9). Plus the
  first conventions-layer slice: the model's locale head is exported and OBEYED (fr mask).
- **Open gaps:** FR venue/region ([#330](https://github.com/sister-software/mailwoman/issues/330)),
  intersections ([#487](https://github.com/sister-software/mailwoman/issues/487)).

Capability-arena truth (whole-parse, vs the v0 rules system): rules still win on
clean/canonical input, the neural parser wins decisively on noisy/degraded input (+21pp),
both are weak on rare edge formats. The
[arbitration layer (#478)](https://github.com/sister-software/mailwoman/issues/478) exists to
make the pipeline take the better of both per component.

## Confidence

Every node carries a confidence score; pass a calibrator (isotonic, shipped per-locale in the
weights bundle as `calibration.json`) to make those scores honest probabilities — see
[Confidence calibration](./concepts/confidence-calibration.mdx) for what the numbers mean.

## Browser demo

The live demo at [mailwoman.sister.software/demo](https://mailwoman.sister.software/demo) runs the
int8 model (29.8 MB) via `@mailwoman/neural-web` plus a **byte-range global gazetteer** — the
FTS-free "candidate" table (~530 MB, queried over `sql.js-httpvfs`; a resolve touches ~12 range
fetches regardless of file size) with global admin coverage + **US + EU postcode resolution** and a
debounced FST place-autocomplete in the search box. US **street-level (rooftop)** resolves for
CA/NY/MI/DC via per-state situs/interp shards. Artifacts are served byte-range from R2. The demo
calls the classifier's argmax path directly; joint reconcile is not active in the demo.

## Model line (June 2026)

The shipped weights are **4.15.0** — `v1.9.3a3-anchor-absorption`, the latest step in a coordinate-first arc. The lineage stacks three single-lever moves, each held to the **shipped assembled coordinate** (not label-F1): v4.11.0 (`v1.8.0-fr-admin-split`) taught the locality↔adjacent-admin split on non-US formats (FR coord p50 42 → 2.2 km); v4.14.0 (`v1.9.2-multilocale-au`) added AU word-order via G-NAF (AU @25km 65 → 87); v4.15.0 closes the #220/#723 postcode-anchor-pollution bug — a leading 5-digit house number that looks like a ZIP is now labeled `house_number` with the anchor ON (`SLICE-H` 20 → 100), at zero coordinate cost (oa-resolver-eval, 2000 real OA-US points: whole-US locality 97.7 → 97.8, p50 3.4 km flat). The trade is a coordinate-invisible −2 us.postcode label-F1 on the rare VT-E911 leading-postcode case. The parity campaign's remaining gaps are quality gradients, not missing capabilities. Next: the post-parity agenda — arbitration ([#478](https://github.com/sister-software/mailwoman/issues/478)), geocoder table stakes (interpolation, reverse), and the char-offset corpus format (the structural cure for what the span bridge contains). The ordered
work queue to the full geocoder is
[epic #488](https://github.com/sister-software/mailwoman/issues/488).

## What is not supported today

- Rooftop / address-point geocoding **nationwide** — available for **CA/NY/MI/DC** (per-state
  situs + interp shards, [#476](https://github.com/sister-software/mailwoman/issues/476) /
  [#483](https://github.com/sister-software/mailwoman/issues/483)). The 50-state layer is built
  ([#567](https://github.com/sister-software/mailwoman/issues/567)) but only those four are hosted;
  the national rollout is [#735](https://github.com/sister-software/mailwoman/issues/735). Other
  states resolve to the admin centroid.
- Reverse geocoding ([#484](https://github.com/sister-software/mailwoman/issues/484)). _(Place
  autocomplete IS now shipped — the demo search box + the CLI `autocomplete` command,
  [#190](https://github.com/sister-software/mailwoman/issues/190) /
  [#587](https://github.com/sister-software/mailwoman/issues/587).)_
- Multi-locale ensemble (en-US + fr-FR weights in the same runtime).
- Korean addresses (no open data path). Japanese addresses **are** supported via the
  postcode-route resolver (Geographic Rule Engine).
- A supported server API — `mailwoman/server` ships an experimental Express server
  (`/api/parse`, `/api/resolve`); production service hardening is
  [#485](https://github.com/sister-software/mailwoman/issues/485).
