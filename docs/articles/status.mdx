# Status

This page is the single source of truth for what works in Mailwoman right now. For where the
project is going — locale tiers, workstreams, standing invariants — see
[Current scope & roadmap](./plan/SCOPE.mdx); the Archive's plan directory is history, not the
roadmap.

:::info[Verified as of 2026-07-15 — release 6.3.0]
The current npm release is **6.3.0** (`npm view mailwoman version`; `git tag` confirms `v6.3.0`), a
minor that promotes the **v264 country-softguard** model. 6.2.1 was the patch that wired the browser
country channel into `@mailwoman/neural-web`; the model first shipped as **6.2.0** — the v263
country-channel line. The earlier 6.1.0 minor promoted the v261 span-boundary line, and 6.0.0 was the
breaking, code-only Hono API-surface cutover (all four HTTP surfaces — the native `mailwoman serve`
plus the photon/nominatim/libpostal drop-ins — on one shared Hono stack, the express server deleted
with no shim). See [Releases & capabilities](./releases.mdx) for the rows.

**The current model is v264** (`v2.6.4-country-softguard`,
[#1104](https://github.com/sister-software/mailwoman/issues/1104)) — an init_from fine-tune of v263
with **one variable**: `country_ambiguous_scale` 1.0 → 0.5. v263's dedicated country soft-feed channel
(a 2-dim `[country_surface, country_ambiguous]` additive feature with its own projection + confidence,
immune to the near-postcode gazetteer suppression — the atlas analogue of Pelias's
`WhosOnFirstClassifier`) carried a **hard** homograph guard: the `country_ambiguous` bit fully
suppressed country surfaces that are also US states / cities / common words (Georgia, Jordan, Turkey).
That over-suppressed country broadly. v264 softens the guard — the ambiguous dim is scaled 0.5 before
`country_projection` via a forward-pass buffer that **bakes into the exported ONNX** (no lexicon,
inference, or browser change). The result **strictly dominates v263** with no offsetting trade:
country-homograph F1 **82.6 → 85.1**, golden WOF-admin country recall **89.3% → 91.1%** (the softening
helped BOTH distributions — the hard guard was over-firing), real-postal country recall + hallucination
HELD, held-out coordinate z-test PASSES on both locales, aggregate golden label-fails **1101 → 1098**.
It inherits v263's country-channel win over v261's cosmetic gate exception and v261's span-boundary win.
The tokenizer is unchanged, `v0.9.0-multisplice` (73,143 pieces). The v265 (scale 0.25) probe recovered
more of the narrow homograph slice (87.5) but slightly less WOF-admin; 0.5 was chosen for balance. See
the [v264 country-softguard grade](./evals/2026-07-15-v264-country-softguard-result.md) for the full
table.

**The per-tag tables below are frozen at the 2026-07-02 full re-score of the earlier 5.0.0/v4.15.0
line** (the #885 measurement re-anchor). Several model releases have shipped since without a fresh
full re-score (5.1.0 → 5.4.0 → 5.9.0 → 6.1.0 → 6.2.0 → 6.3.0); their measured deltas are narrated
release-by-release — not folded into a new scorecard — on
[Releases & capabilities](./releases.mdx), which is the current per-release source of truth. Treat
the [parity scorecard](./evals/competitive-parity/parity-scorecard-2026-07-02.md) below as a
historical baseline, not the shipped model's live per-tag truth.
:::

The trend, in words:

- **Healthy (2026-07-02 re-score):** postcode (95.0/99.3 US/FR — both sitting exactly at their
  revised gate floors), house_number (90.7/98.1), venue-US (89.1), street (82.3 US, the best it
  has measured), fr.region recovered 16.2 → 48.4 by the multi-locale work.
- **Fixed by the parity campaign:** `unit` 0 → 92.3 on real-OOD designators (the 4.1.0 headline).
- **Shipped by the consolidation (v4.2.0):** locality +12.8, region +10.7, country 0 → 89.8
  (gazetteer soft anchor), affixes now exist (0 → 64.9/48.8 at P≈100), DE 90.9.
- **Closed by the boundary consolidation (v4.4.0):** the scorecard's last empty rows — po_box
  0 → 89.1, cedex 0 → 96.1, intersections 0 → 100 (real-OOD, P ≥ 95) — and the v4.3.0 arena
  debt (perturb 64 → 72, now a gated floor). The span bridge ships as required decode config:
  dotted surfaces ("P.O. Box") are structurally unlabelable in the corpus format; the bridge is
  the containment ([ship gate](./evals/model-versions/2026-06-11-v4.4.0-ship-gate.md) tells the three-battery story).
- **Closed by the relabel (v4.3.0):** the affix ceiling was never capacity — it was a 1,039:1
  label contradiction in the training mix ([#492](https://github.com/sister-software/mailwoman/issues/492) →
  [#511](https://github.com/sister-software/mailwoman/issues/511)). Affixes now 93.6/96.6 (int8,
  real-OOD; 92.2/90.3 on the 193-row NAD-native eval where v4.2.0 scored 18.2/8.9). Plus the
  first conventions-layer slice: the model's locale head is exported and OBEYED (fr mask).
- **Formerly "open gaps," now closed:** the two rows this section used to flag are both resolved.
  FR venue/region ([#330](https://github.com/sister-software/mailwoman/issues/330)) was a
  measurement artifact — the model's FR venue F1 is 89.1+ and région 99.6% in-distribution,
  not the 0%/19% a synthetic multi-script eval implied. Intersections
  ([#487](https://github.com/sister-software/mailwoman/issues/487)) were fixed by the v4.4.0
  real-TIGER intersection shard (0 → 100/100 real, 94.9/96.1 VT golden).

Capability-arena truth (whole-parse, vs the v0 rules system): rules still win on
clean/canonical input, the neural parser wins decisively on noisy/degraded input (+21pp),
both are weak on rare edge formats. An
[arbitration layer (#478)](https://github.com/sister-software/mailwoman/issues/478) was built to
make the pipeline take the better of both per component, but its promotion was falsified by the
coordinate gate ([#685](https://github.com/sister-software/mailwoman/issues/685) — arena +122
label-wins cost coord p50 3.3 → 1069 km and broke the street+house_number geocode precondition
100% → 48%). It stays merged, default-off, research-only.

## Confidence

Every node carries a confidence score; pass a calibrator (isotonic, shipped per-locale in the
weights bundle as `calibration.json`) so a score of 0.90 means right-about-90%-of-the-time — see
[Confidence calibration](./concepts/confidence-calibration.mdx) for what the numbers mean.

## Browser demo

The live demo at [mailwoman.sister.software/demo](https://mailwoman.sister.software/demo) runs the
int8 model (37.6 MB, the v264 line — the softened country channel bakes into the same ONNX the browser
country wiring in `@mailwoman/neural-web` 6.2.1 already feeds) plus a **byte-range global
gazetteer** — the FTS-free "candidate" table (~530 MB, queried over `sql.js-httpvfs`; a resolve
touches ~12 range fetches regardless of file size) with global admin coverage + **US + EU postcode
resolution** and a debounced FST place-autocomplete in the search box. US **street-level
(rooftop)** resolves nationwide — all 50 states + DC + VI — via per-state situs/interp shards
(124.9M address points, [#735](https://github.com/sister-software/mailwoman/issues/735)); FR
rooftops resolve via the hosted BAN situs shard with a street-tier fallback (shipped 5.10.0, see
[Releases & capabilities](./releases.mdx)). Artifacts are served byte-range from R2. The demo
calls the classifier's argmax path directly; joint reconcile is not active in the demo.

## Model line (July 2026)

The shipped weights are the **v264 line** (`v2.6.4-country-softguard`, promoted 6.3.0 —
[#1104](https://github.com/sister-software/mailwoman/issues/1104)). Model releases
shipped between the 5.0.0/v4.15.0 line (documented in past versions of this page) and today, each
held to the **shipped assembled coordinate or a targeted repro**, not raw label-F1: v5.1.0
(`v0.6.0-bsplice` tokenizer splice) fixed Slavic diacritic fragmentation — CZ/PL/SK/SI wrong-city
roughly halved, US byte-identical; v5.2.0 (`v0.7.1-nsplice`) extended the splice to Nordic
diacritics (SE resolve 83.8 → 90.2%); v5.3.0 (`v2.2.0-full-family`, from scratch) fixed the FR
street fragmentation the v5.2.0 splice had shipped ungated (bare 16/40 → 34/40, anchored 13/40 →
37/40); v5.4.0 (`v2.3.0-nl-postcode`, fine-tune) fixed a Dutch digits-first postcode misread that
was pulling `1012 LG Amsterdam` into the US situs tier; v5.9.0 (`v2.4.1-fr-nsplice-ft`, fine-tune)
gave the merged FR diacritic pieces gradient after a training-free vocab splice was falsified for
regressing FR streets, cutting the accent mangle rate 23.5% → 17.7%; v6.1.0/v261
(`v2.6.1-span-boundary-full`, fine-tune off the stable v257 fragment-campaign line) trained a
span-boundary auxiliary head to shrink the region→street boundary-absorption class (US region→street
flips 5→2 vs v257, parity street +13pp vs v241), shipping through a documented cosmetic gate
exception on golden country recall ([#1104](https://github.com/sister-software/mailwoman/issues/1104) —
real-postal country recall and precision are unaffected); **v6.2.0/v263**
(`v2.6.3-country-channel`, fine-tune off v261) turned that exception into a fix — a dedicated country
soft-feed channel recovered golden country recall 84.8→89.3% (clearing the 88.6% v241 bar) while the
held-out coordinate held flat on both locales, carrying a coordinate-invisible region per-tag
trade (6.2.1 then wired the channel into the browser runtime); and **v6.3.0/v264**
(`v2.6.4-country-softguard`, fine-tune off v263) softened that channel's homograph guard
(`country_ambiguous_scale` 1.0→0.5, baked into the ONNX) — strictly dominating v263 with no trade:
country-homograph F1 82.6→85.1, golden WOF-admin country recall 89.3→91.1%, aggregate golden
label-fails −3, coordinate PASS both locales. Full per-release evidence lives on
[Releases & capabilities](./releases.mdx), which this page points to for anything shipped after
the last full re-score. The ordered work queue to the full geocoder is
[epic #488](https://github.com/sister-software/mailwoman/issues/488).

## What is not supported today

- Multi-locale ensemble (en-US + fr-FR weights in the same runtime).
- Korean addresses (no open data path). Japanese addresses **are** supported via the
  postcode-route resolver (Geographic Rule Engine).

Three items that used to live on this list have shipped and are covered elsewhere on this page:
nationwide US rooftop geocoding (all 50 states + DC + VI, see **Browser demo** above,
[#735](https://github.com/sister-software/mailwoman/issues/735)), reverse geocoding (`mailwoman
reverse`, see the **Runtime pipeline** table above, [#484](https://github.com/sister-software/mailwoman/issues/484)),
and the production-hardened server API (batch, observability, `RemoteResolver`, versioned data
switchover — shipped v4.7.0, [#485](https://github.com/sister-software/mailwoman/issues/485); the
6.0.0 Hono cutover retired the last express-era code path, see
[Releases & capabilities](./releases.mdx)).
