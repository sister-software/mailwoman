# The v1 implementation plan — historical

**This is a design record, not the live plan.** The current scope declaration — locale tiers,
workstreams, standing invariants, where the roadmap lives — is **[Scope — what mailwoman is
now](./SCOPE.mdx)** ([#886](https://github.com/sister-software/mailwoman/issues/886) marked this
directory historical on 2026-07-02).

What follows below this note is the plan that drove the neural classifier work from Phase 0
through the release train, preserved as written. It declared a US + France v1 with Japanese as a
Phase 6 stress test; the shipped reality (a multi-locale geocoder with drop-in API surfaces and a
record-matching workstream) outgrew it — which is what plans are for. Read it, and the
`phases/*` + `reference/*` files it indexes, for design rationale and the reasoning behind
decisions that still bind (`reference/SCHEMA.mdx` remains the live source of truth for the
`ComponentTag` union; [CONTRIBUTING_MODEL_WORK](./CONTRIBUTING_MODEL_WORK.mdx) remains the live
model-work runbook).

## The plan as written (2026-05, status notes through 2026-06-11)

TypeScript-first, deployed as a new classifier that progressively replaces rule-based classifiers
as confidence metrics justify (Ship of Theseus).

**Status (last edited 2026-06-11):** Phases 0 through 4 shipped and superseded by the release
train. The live roadmap moved to the
[Geocoder v1 — Daily Queue project board](https://github.com/orgs/sister-software/projects/4)
(epic [#488](https://github.com/sister-software/mailwoman/issues/488)), plus the dated eval
reports in `docs/articles/evals/`.

- The capacity question this plan deferred ("bigger model later?") was answered by measurement:
  a 48M widening reproduced the 29M affix ceiling exactly — the binding constraint was a
  corpus label contradiction, not parameters
  ([#492](https://github.com/sister-software/mailwoman/issues/492) →
  [#511](https://github.com/sister-software/mailwoman/issues/511)).
- Architecture + per-stage contracts: `reference/ARCHITECTURE.mdx`, `reference/STAGES.mdx`; the
  conceptual map is [the knowledge ladder](../understanding/our-approach/the-knowledge-ladder.mdx).

### Hard constraints (as declared for v1)

- **TypeScript-first.** Inference runtime is `onnxruntime-node`. Training is allowed in Python
  but is not shipped to npm.
- **Coexistence, not replacement.** The neural classifier is additive. Rule classifiers are not
  deleted until per-component metrics justify retirement.
- **Locale scope (v1):** US + France. English + French. Japanese is a deliberate Phase 6 stress
  test of the architecture, not v1. _(Superseded — see [Scope](./SCOPE.mdx) for the real list.)_
- **Staged components:** coarse (country/region/locality/postcode) first, street second, venue
  third. Each stage ships.
- **No new ML in places where rules are already correct.** Postcodes are a regex problem. Don't
  burn a model on them.
- **Schema-first.** Touching `reference/SCHEMA.md` requires a written rationale in the commit.
  Downstream code keys off it.

### What good looked like

By end of Phase 3: `@mailwoman/neural@0.1.0` on npm, loading a quantized ONNX model, emitting
`ClassificationProposal` objects through the existing solver, beating rule-based Mailwoman on the
held-out golden set for `country` and `region`. By end of Phase 6: the same architecture supports
Japanese addresses with schema additions but no core refactor. Both happened — the second via the
postcode-route resolver rather than parser training, which is its own validation of the
schema-first bet.
