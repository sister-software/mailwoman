# #727 stage-2 Phase 2 — export verdict: all three gates PASS

**Gates (pre-registered in `2026-07-15-727-stage2-phase2-export.md`):**

> 1. ONNX `span_scores` ≈ torch (atol 1e-3 fp32) on random inputs — parity or no ship.
> 2. int8 size delta vs v264's 39.8 MB within +1 MB.
> 3. Fetching ONLY `logits` from the span-enabled graph costs ≤5% latency vs the v264 graph.

## Results

| gate                   | bar     | measured                                                   | verdict  |
| ---------------------- | ------- | ---------------------------------------------------------- | -------- |
| 1. ONNX↔torch parity   | ≤ 1e-3  | worst **7.3e-06** over 5 real addresses (fp32)             | **PASS** |
| 2. int8 size           | ≤ +1 MB | 40,065,060 vs 39,838,216 = **+0.22 MB (+0.57%)**           | **PASS** |
| 3. logits-only latency | ≤ +5%   | **−13.9%** (3.51 ms vs 4.08 ms, CPU ORT, S=128, 200 iters) | **PASS** |

**Honesty note on gate 3:** the negative delta is NOT the span head making inference faster — the two
graphs come from different export runs, and toolchain/graph-optimization drift plausibly explains a
14% gap. The defensible claim is only the gate's own: _a runtime that never fetches `span_scores`
pays nothing measurable._ ORT prunes the unfetched branch, as designed.

**The Phase-3 decode budget:** fetching `logits+span_scores` costs **+0.75 ms** over logits-only
(CPU, S=128). That is the input-side cost the JS k-best decoder inherits; the decode itself is
JS-side on ~100–200 pruned spans (consult: "decode ~free").

## What changed in code

- `export_to_onnx`: the five wrappers gain a `with_spans` toggle mirroring `with_locale`;
  `output_names` built from both flags. Span-less export byte-compatible (tested: `["logits"]`).
- `package_weights.export_semi_crf_transitions()` — the JSON sidecar, `None` for span-less models,
  following the `export_crf_transitions` precedent (transitions are decode-time data, not graph).
- `sync_src_727_stage2` verify extended to 10 checks (export + sidecar presence) — a stale
  container-side export would silently produce a graph without `span_scores`.
- `onnxscript==0.7.0` added to the `[train]` extra (the dynamo exporter's dep; pinned to the Modal
  image's version). Collateral: uv re-resolve bumped trackio 0.27.0→0.30.3.

## What Phase 3 consumes

1. The int8 graph's `span_scores` output — shape `(B, S, 8, 17)`, fetched by name.
2. `semi-crf-transitions.json` — decode-time transition grammar, axis read from the file.
3. The stage-1 boundary head is NOT exported (training-only, unchanged) — Phase 3's span pruning uses
   the span scores themselves (top-K per position) rather than the aux head's probs; revisit only if
   the candidate set blows past ~200.

Phase 3 (JS/WASM k-best in `neural/` + `neural-web/`) is unblocked. The decoder's reference
implementation is `mailwoman/eval-harness/oracle-k.ts` (same recurrence, learned scores replacing the
summed-BIO stand-in) and the Python `SemiMarkovCRF.decode` (brute-force verified).
