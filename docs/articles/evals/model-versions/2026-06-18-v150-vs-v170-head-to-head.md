# v1.5.0 vs v1.7.0 — promote head-to-head (2026-06-18)

## TL;DR

v1.7.0 is **not** a promote over the production default v1.5.0. On assembled coordinates — the
metric we ship — the two are **flat on both US and FR**. Measured with the production-faithful
(anchor-on) harness, v1.7.0's per-tag F1 is a small us.locality gain offset by us.street, fr.region,
and country regressions — not a net improvement even at the label level. **Recommendation: HOLD
v1.5.0.**

## The baseline correction (the reason this re-eval happened)

The night-shift gate compared v1.7.0 against **v1.5.1** — a falsified weight-6.0 experiment that
was never promoted. The production default is **v1.5.0** (model-card 4.6.0; confirmed by md5
`4674d3…` == `model-v150-step-40000-int8.onnx`, the fr-order recovery model). All deltas below are
vs v1.5.0.

**Update (post-localadmin scoreTree fix, `9f986ec3`):** these locality-match figures were UNDER-COUNTED. The eval's `scoreTree` selected only the bare `locality` placetype and discarded the `localadmin` (New England town) resolutions the resolver had landed correctly. Re-measured with the corrected metric (crediting the resolver's locality∪borough∪localadmin group): **v1.5.0 US locality-match 83.9% → 97.8%** (SD 62.5→97.5, VT 31.6→92.6); region-match and coords unchanged — the resolver was always landing the right place. The v1.5.0-vs-v1.7.0 comparison stays FLAT (the fix is parser-agnostic), so the promote verdict is unchanged — but the rural "coverage gap" was mostly this scoring artifact, not real under-resolution.

**Update 2 (situs cascade, `dd3628da`):** the coord figures above are the **admin-centroid** tier — `oa-resolver-eval` resolved to the resolved place's centroid and never wired the situs cascade the geocoder actually ships (`mailwoman/geocode-core.ts`). Graded through the production cascade (`--cascade`, per-state address-point + interpolation shards over the #567 layer) the same 10k US rows resolve to **p50 0.0 km, p90 1.0 km, 85.9% within 100 m** (79.8% address_point / 8.2% interpolated / 12.0% admin). Model-independent — both versions run the same resolver, so the promote verdict is unchanged — but it retires the "coordinate bottleneck" framing below: the shipped US coordinate is meter-grade, not admin-centroid. Full write-up: [`2026-06-18-situs-cascade-eval.md`](../resolver-geo/2026-06-18-situs-cascade-eval.md).

## FR — assembled coordinates (corrected)

An earlier run resolved under 10% of FR — an eval-flag error: `oa-resolver-eval`'s
`--default-country` defaults to `US`, and the FR sample carries no country token, so every FR lookup
was filtered to US places (landing the addresses in America, p50 ~7200 km). With
`--default-country FR` the resolver handles FR fine (`admin-global-priority.db` holds 114k FR places).
Re-run correctly:

| ------------------- | -----: | -----: | ---: |
| locality-match      |  76.3% |  76.3% | flat |
| coord p50           | 1.5 km | 1.5 km | flat |
| coord p90           | 5.3 km | 5.3 km | flat |

Flat. v1.7.0's FR work doesn't reach the FR coordinate any more than its US work reaches the US
coordinate. (region-match is 0% for both — the FR sample has `region: null`, so there's no gold

## country-homograph (n=54, real homograph addresses)

| tag         | v1.5.0 | v1.7.0 |        Δ |
| ----------- | -----: | -----: | -------: |
| country F1  |   83.3 |   80.9 | **−2.4** |
| region F1   |   90.9 |   93.8 |     +2.9 |
| locality F1 |   82.0 |   95.3 |    +13.3 |

The −2.4 is **one row**, not a systematic regression (diagnosed 2026-06-18). The set is 54 rows, 27
with a gold country; v1.7.0's miss set is v1.5.0's plus exactly one: `Avenida Arequipa, Lima 15046,
Peru` — v1.7.0 keeps street/locality/postcode identical but demotes the trailing "Peru" to O (zero
reverse flips; on n=27, one row ≈ 3.7 F1 points). The trigger is a narrow conjunction — a strong
US-locality **homograph** token ("Peru" = Peru IN/IL; "Lebanon" = Lebanon TN/NH, both literally US
rows in this set) + trailing position + a preceding numeric postcode. Swap "Peru" for
Chile/Bolivia/Ecuador/Colombia/Brazil/France/Jordan in the same frame and v1.7.0 is correct; drop
the postcode and "Peru" is correct; the structurally-identical Jordan/Greece rows pass. It is NOT a
general "trailing-country-after-postcode" weakness — v1.7.0 actually splits the no-postcode
trailing-country rows (Beirut/Lebanon, Naples/Italy) _better_ than v1.5.0. So the HOLD rests on the
flat assembled coordinate vs v1.5.0, not on this one homograph row. (The 83.3 the night called a
"stale floor" is exactly v1.5.0's score.)

## Golden per-tag F1 (production-faithful, anchor-on)

The first per-locale-f1 pass reported v1.7.0 catastrophically worse (us.country 0, fr.house_number
58.5) — a **harness bug**: `per-locale-f1.ts` fed the model **anchor-off**, which is
out-of-distribution for these anchor-trained STAGE3 models and selectively collapses the admin tags
(country/region/locality/postcode) while morphology tags survive. Fixed — anchor feed is now
default-on (commit `d7b51748`); both v1.5.0 and v1.7.0 crater identically anchor-off and recover
identically anchor-on. The production-faithful numbers:

| tag             | v1.5.0 | v1.7.0 |        Δ |
| --------------- | -----: | -----: | -------: |
| us.locality     |   77.9 |   80.5 |     +2.6 |
| us.region       |   90.5 |   91.3 |     +0.8 |
| us.street       |   80.2 |   76.2 | **−4.0** |
| us.house_number |   98.3 |   97.3 |     −1.0 |
| us.country      |   68.4 |   68.9 |     +0.5 |
| fr.house_number |   99.3 |   99.3 | **flat** |
| fr.region       |   41.8 |   35.7 | **−6.1** |
| fr.country      |   62.7 |   63.0 |     +0.3 |

**The night's headline "fr.house_number +10.2" was an anchor-off measurement artifact** — anchor-on,
both v1.5.0 and v1.7.0 score 99.3. Measured correctly, v1.7.0 is a small us.locality gain (+2.6)
offset by us.street −4.0, fr.region −6.1, us.house_number −1.0, and country −2.4. It is not a net
improvement even at the label level, and none of it reaches the assembled coordinate (flat on both
locales).

## Recommendation

1. **HOLD v1.5.0; do not promote v1.7.0.** Coordinate-flat on both US and FR; a label-level wash whose
   only "regression" is a single homograph row (Lima/Peru, see country-homograph above). There is no
   version of this comparison that favors the swap.
2. **The shipped US coordinate is already meter-grade — the "bottleneck" was the eval grading the admin
   centroid** (see Update 2 + the situs-cascade doc). Through the production cascade, 88% of US
   addresses resolve at the address-point/interpolation tier (p50 0.0 km, 85.9% within 100 m). The
   model has caught up to its database on both locales; the remaining coordinate gain is point-data
   coverage for the ~12% admin tail, not another retrain and not broader gazetteer breadth.
3. **v1.7.1 (recover street) is not worth the GPU** — v1.7.0 is already coord-flat with v1.5.0.
4. **Both tooling bugs are fixed/diagnosed:** `per-locale-f1.ts` now feeds the anchor by default
   (`d7b51748`); FR coord evals need `--default-country FR` (no code change).

## The row-level read (2026-06-19) — what the −4.0 / −6.1 actually are

The per-tag deltas above were aggregate-only. Pulled to the row level (`MAILWOMAN_DUMP_MISS_TAG`,
v1.5.0 vs v1.7.0 int8, anchor-on; dumps under `/tmp/reg/*.miss`), they are softer than the macro
implied — and they explain mechanically why the assembled coordinate stayed flat.

**us.street −4.0 = 96 newly-missed rows, 34 fixes (net +62 on 2,660).** The dominant class is not
damage to street _names_ — it is span-boundary clipping at the edges:

- **51/96 (53%) trailing-directional-drop** — `Woodland Ave NE → Woodland Ave`, `3rd St NW → 3rd
St`, `Hillcrest Dr NE → Hillcrest Dr`. v1.7.0 cuts the span before the trailing `NE/NW/SE/SW` and
  re-tags it `unit`. This is exactly the #723 directional class, and the directional street-key fold
  shipped 2026-06-19 recovers it at geocode (+1.8 address-point / +1.7 within-100m). The
  boundary-stress shard over-corrected span boundaries the other way.
- **14 leading-ZIP-mangle** — `05770 VT DELONG LN → DELONG`, `58703 Lakeside St → St`. Reordered
  synthetic golden rows (ZIP-first), not the street-first order production geocodes.
- **8 numbered-digit-eat** — `74TH ST SE → 4TH ST SE`, `158TH AVE NE → 8TH AVE`. A real
  numbered-street boundary error, thin slice.
- **20 other**, incl. a few genuine misfires (`Wyoming 82435, Rd 21 → "Wyoming"`) and tokenizer
  noise (`Ski Tow Rd → "S ki Tow Rd"`).

**fr.region −6.1 = only 17 rows, OOD-concentrated, zero coordinate impact.** Every regression row is
a bare `Locality, Département` FR row (Creuse/Lozère) where v1.7.0 drops the region to null
(`Thauron, Creuse → ∅`) or, on the accented `Lozère`, emits a broken trailing subword — `Montredon,
Lozère → "ère"` (the model splits the diacritic, putting `Loz` in locality and `ère` in region; this
is **not** v1.7.0-specific — v1.5.0 does it on 4 rows too; filed as its own bug). Inputs skew OOD
(CJK localities, Uzbek). The FR coordinate sample is `region: null`, so this label slice never
touches a coordinate (FR coord flat 1.5 km both).

**Net:** reading the rows makes the HOLD firmer as "a wash with no upside" and weaker as "v1.7.0
broke things." It is coordinate-flat because the one systematic street regression is the directional
class the #723 fold recovers, and the FR piece is a 17-row OOD diacritic slice. Verdict unchanged;
the reason is "no gain," not "new damage."
