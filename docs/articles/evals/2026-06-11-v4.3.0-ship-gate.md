## The FR digit-split corrective (conventions mask, #478 slice 1)

First gate run FAILed fr.postcode 99.3/99.5: 6 misses, of which 4 NEW — leading-postcode
digit-splits, present at 20k and 40k (a stable property of the relabeled mix: standalone numeric
street names made an orphan postcode digit a credible street start; `street_suffix: "Rue"` showed
USPS clue leakage into French parses — RUE is genuinely a Pub-28 suffix variant). The other 2 are
Montréal po_box rows both models fail (deferred lever, not a regression).

Fix: export the PR3 locale head (`locale_logits`, trained at ~98% accuracy, previously
unreadable at inference) + `codex/address-system-conventions.ts` (fr: affix tags ungrammatical
per NF Z 10-011; postcode shape `^\d{5}$`) + a hard emission mask before Viterbi + the
conventions shape enabling the existing snap-only postcode repair. Detection threshold 0.8 — the
mask never fires on a guess; non-fr parses are byte-identical (verified: every US/DE number above
is unchanged from the pre-conventions run). All four digit-split rows decode the full postcode.

Row-level diff says the perturb drop is DOMINATED BY ONE CLASS: the "glue" perturbation —
region+postcode fused (`NY14201`, `IL60602`, `TX77002`). v4.2.0 split the glued token (region NY

- postcode 14201); v1.1.0 swallows it whole as `postcode: "NY14201"` and loses the region
  (~24–31 of the 40 flips). The remainder are street-boundary wobbles: post-directional `NW` →
  locality, `Main St Apt` absorbing the designator, a venue bleeding into street. This is the SAME
  digit/boundary-instability family as the FR digit-split — the relabel pinned the affix boundary
  hard and loosened adjacent boundary classes the corpus does not pin. The US glue class has no
  conventions row to catch it (and a snap repair would fix the postcode but not recover the
  region), so the durable fix is training-side: a glue-class augmentation in the next corpus pass.
  Follow-up issue filed.
