# v1.5.0 vs v1.7.0 — promote head-to-head (2026-06-18)

## TL;DR

v1.7.0 is **not** a clean promote over the production default v1.5.0. On the decisive
metric — assembled US coordinate accuracy — the two are indistinguishable. v1.7.0 carries a
real country-homograph regression. Its golden-F1 gains do not reach the assembled coordinate,
and its FR value is unconfirmable at the coordinate level with the current resolver.
**Recommendation: HOLD v1.5.0.**

## The baseline correction (the reason this re-eval happened)

The night-shift gate compared v1.7.0 against **v1.5.1** — a falsified weight-6.0 experiment that
was never promoted. The production default is **v1.5.0** (model-card 4.6.0; confirmed by md5
`4674d3…` == `model-v150-step-40000-int8.onnx`, the fr-order recovery model). All deltas below are
vs v1.5.0. The night's "+5.1 locality / fixes a failing fr.house_number floor / stale country floor"
framing was all measured against the wrong, worse model.

## US — the decisive comparison (OpenAddresses, 2000 real gov points, full pipeline)

| metric | v1.5.0 (int8) | v1.7.0 | Δ |
|---|--:|--:|--:|
| locality-match | 83.9% | 83.8% | flat |
| region-match | 99.9% | 99.9% | flat |
| coord p50 | 3.3 km | 3.3 km | flat |
| coord p90 | 10.7 km | 10.7 km | flat |

v1.7.0's locality work **does not move the assembled US coordinate.** Per-state, the misses are
rural-gazetteer coverage — SD 62.5%, VT 31.6% locality-match, identical for both models — not model
tagging. (v1.7.0 US coords are fp32; int8 would be ≤ that, so "flat" is robust to the ~0.1pp quant
noise.)

## country-homograph (n=54, real homograph addresses)

| tag | v1.5.0 | v1.7.0 | Δ |
|---|--:|--:|--:|
| country F1 | 83.3 | 80.9 | **−2.4** |
| region F1 | 90.9 | 93.8 | +2.9 |
| locality F1 | 82.0 | 95.3 | +13.3 |

A real country regression — v1.7.0 misses one more leading/trailing country token ("United States,
…", "France, …", "…, Georgia"). The 83.3 the night called a "stale floor" is exactly v1.5.0's score.

## Golden per-tag F1 — measurement flagged, not trusted

`per-locale-f1.ts` produces internally-inconsistent output for the v1.7.0 model (us.country 0,
fr.country 0, fr.house_number 58.5) that contradicts the **same model's** production-pipeline result
(region-match 99.9%) and homograph eval (country 80.9). This is a harness bug specific to that script
+ this model — not a model regression — and the numbers are withheld pending a clean re-run via a
different harness (`eval-error-analysis.ts`). The night's gate (`boundary-stress-gate.ts`, unaffected)
reported v1.7.0 us.locality 80.0 / fr.house_number 94.9 vs v1.5.0 77.3 / 87.4 — real model-level
gains that, per the US coordinate result above, **do not reach the assembled coordinate.**

## FR coordinates — unmeasurable today

The default resolver shards resolve under 10% of FR (region-match 0%, p50 7200 km) for **both** models —
a resolver/DB coverage gap, not a model signal. v1.7.0's FR value stays a label-F1 claim.

## Recommendation

1. **HOLD v1.5.0; do not promote v1.7.0.** No coordinate-level case, plus a country regression.
2. **The US coordinate bottleneck is the gazetteer/resolver, not the model** — rural-state coverage
   (SD/VT), identical across model versions. The next high-leverage lever is data/resolver coverage,
   not another retrain.
3. **v1.7.1 (recover street) is not worth the GPU** — v1.7.0 is already coord-flat with v1.5.0, so a
   street-recovery refine won't change the US coordinate picture.
4. **File two tooling issues:** `per-locale-f1.ts` mis-decodes the v170 model; the FR resolver
   coverage gap blocks FR coord evals.
