# Try it: the POI intent tester

Type a query below and watch Mailwoman's kind classifier decide whether it's a POI query or an address, then extract the subject ‚Äî a taxonomy category (`hospital`, `drinking fountain`) or a chain brand (`chevron`, `applebee's`) ‚Äî split off the location anchor, and render an OverpassQL export you can paste into [Overpass Turbo](https://overpass-turbo.eu/).

The intent detection runs entirely in your browser with no network request: it's the same `matchPOISubject` + `createKindClassifier` + OverpassQL emitter Mailwoman ships as a library ‚Äî see [`@mailwoman/kind-classifier`](https://www.npmjs.com/package/@mailwoman/kind-classifier) and [`@mailwoman/poi-taxonomy`](https://www.npmjs.com/package/@mailwoman/poi-taxonomy). A brand subject resolves to its Wikidata QID from the same brand table the runtime uses. Hitting "Search live" on a category result adds byte-range HTTP requests against the published gazetteer + POI layer ‚Äî still no server-side code, no model weights, but not offline either.

## Reading the result

- **Kind badge** ‚Äî the Stage 2.5 kind classifier's verdict (`poi_query`, `structured_address`, ‚Ä¶) with its confidence. Expand it to see the runner-up kinds.
- **Category chip** ‚Äî which taxonomy category the subject phrase matched, when the kind is `poi_query`.
- **Brand chip + QID** ‚Äî when the subject is a chain brand, the brand's canonical name plus a linked Wikidata QID (`Q319642` = Chevron). Brands resolve by that QID against the layer's `brand_wikidata` index, not by category ‚Äî so they carry no OverpassQL export and no OSM-tag k-ring.
- **build-local badge** ‚Äî this category's data lives only in ODbL sources (OpenStreetMap). Mailwoman ships the builder for the local layer, not the data itself ‚Äî see [Amenity queries](./amenity-queries.mdx) for why that split exists.
- **anchor** ‚Äî the text left over after the subject phrase is split off (`near Springfield`, `, New York`). This is what the runtime pipeline would hand to the address parser to resolve a location constraint.
- **OverpassQL export** ‚Äî a ready-to-run query against the category's OSM tag, scoped to the anchor when one resolved. Overpass is an export target, not a serving backend: Mailwoman prints the query, it never runs it for you.
- **Search live** ‚Äî for a category result with an anchor, range-loads the published `poi.db` (13.68M Overture-places rows, US/CA/MX/FR) straight in your browser via sql.js-httpvfs and shows real, distance-ranked results ‚Äî no server, no model weights, just byte-range reads against the same admin gazetteer + POI layer the CLI's `mailwoman poi` and the `mailwoman_poi_search` MCP tool query. It's a k-ring walk capped a few hundred meters out per click, category-only, and only as good as the anchor: an unresolvable place name shows "couldn't place" rather than a silent empty list. A `build-local` category never offers the button ‚Äî its data isn't published, by design. Live search stays category-only on purpose: a brand-wide fetch by QID has to hydrate every one of a brand's globally-scattered rows from the 3.9 GB layer, and byte-range makes that pathological (even a 25-row brand costs ~13 s and 60 range requests; the big ones time out). That path is a server-side capability ‚Äî the Node `POILookup` does the same QID fetch in under a millisecond ‚Äî so brand subjects here stay intent-only.

## Why "hospital, 350 5th Ave, New York" doesn't become a POI query

Try that preset. The lexicon matches `hospital` as a subject, but the remainder ‚Äî `350 5th Ave, New York` ‚Äî leads with a house number. That's the signature of a venue-led full address ("the hospital located at this specific address"), not a category search, so the classifier's `structured_address` score wins and the query falls through to the "parses as an address" state instead of an Overpass export. This is the same guard [the parser uses in production](../our-approach/the-staged-pipeline.mdx) to keep full addresses from getting mis-routed into the POI branch just because they happen to start with a category word.

## What isn't here yet

- **Landmark subjects.** The lexicon matches categories (`hospital`, `fire_hydrant`, `drinking_water`, ‚Ä¶) and chain brands (`chevron`, `applebee's`, from the QID-keyed brand table), but bare landmark names ("Eiffel Tower") aren't in the phrase table yet. See [Landmark queries](./landmark-queries.mdx) for the target shape; [Franchise queries](./franchise-queries.mdx) covers how the brand path works end to end.
- **Live brand search in the tester.** Brand subjects resolve their QID here (intent-only), but the live "Search live" step stays category-only ‚Äî a brand-wide byte-range fetch over the published layer is pathological (see the Search-live bullet above). The CLI and the `mailwoman_poi_search` MCP tool do brand-exact lookups server-side against a built `poi.db` in sub-millisecond time.
- **Free-text (FTS5) name search.** The CLI + MCP tool also expose FTS5 name search against `poi.db`; this browser tester stays intent + category-k-ring, matching the runbook's multi-hop exclusion for the demo.
