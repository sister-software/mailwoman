# Switching from libpostal

[libpostal](https://github.com/openvenues/libpostal) is the address parser everyone links against — a C
library with a CRF model and a 2 GB data download, with bindings you have to build for your language.
Mailwoman exposes the same `parse` / `expand` contract over HTTP (or as a JS library), backed by a
calibrated neural tagger instead of the CRF.

## One command

```bash
npx @mailwoman/libpostal serve --port 8081
```

```bash
curl "http://localhost:8081/parse?query=1600 Pennsylvania Ave NW, Washington DC 20500"
# [{"label":"house_number","value":"1600"},{"label":"road","value":"Pennsylvania Ave NW"},
#  {"label":"city","value":"Washington"},{"label":"state","value":"DC"},{"label":"postcode","value":"20500"}]
```

## Endpoint map

| libpostal              | Mailwoman          | Notes                                             |
| ---------------------- | ------------------ | ------------------------------------------------- |
| `parse_address(addr)`  | `POST/GET /parse`  | ordered `[{label, value}]`, libpostal's label set |
| `expand_address(addr)` | `POST/GET /expand` | normalized forms (see the gaps below)             |

The `ComponentTag` → libpostal label mapping is close to 1:1 (`street` → `road`, `locality` → `city`,
`region` → `state`, `postcode` → `postcode`, …); unmapped tags pass through unchanged.

## What's different

- **A neural tagger, not a CRF.** The model is calibrated, so you also get [a confidence you can route on](./confidence-calibration.mdx)
  — and the same engine geocodes, so parse and resolve share one pipeline.
- **Lowest-dependency drop-in.** `/parse` needs no gazetteer at all; it's just the model.

## What's missing

- **`/expand` returns one alternative, not many.** Mailwoman's normalization is deterministic, so it
  returns the original plus the normalized + abbreviation-expanded form, not libpostal's probabilistic
  multi-variant expansion. If you depend on the full variant set for fuzzy matching, keep libpostal for
  that step.
- **In-process latency.** libpostal links into your binary; Mailwoman is a service (or a JS import). For
  the hottest per-record loops, the in-process C library is still faster.

See [how Mailwoman compares](./how-mailwoman-compares.mdx) for the capability matrix.
