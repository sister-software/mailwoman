Config values (`docs/docusaurus.config.ts`):

- `onBrokenLinks: "throw"` — set explicitly. A broken doc-to-doc link fails the build (and thus
  presumably fails CI's `ci:docs`, though I did not independently confirm CI invokes this same

- `onBrokenAnchors` — **not set**; Docusaurus 3.10's default is `"warn"` (confirmed by reading

- `markdown.hooks.onBrokenMarkdownLinks` — **not set**; default is also `"warn"`
  (`configValidation.js:94`).

## 3. Orphan pages

**Method:** every file under an autogenerated-sidebar directory (`understanding`, `concepts`,
`recipes`, `plan`, `evals`, `retrospectives`, `licensing`) is automatically pulled into that
sidebar's tree by Docusaurus — there's no per-file opt-out in this config (checked: no file in
`articles/` sets `draft: true`, `unlisted: true`, or `sidebar: false` except one, see below, and
that one is separately excluded from the build entirely). So the only place orphans can exist is
among files at the `articles/` root that aren't in the `startHere` explicit array, or in a
directory not wired to any sidebar at all.

**`articles/` root inventory** (non-asset files): `api.mdx`, `contributing.md`,
`getting-started.mdx`, `releases.mdx`, `sotm-2026-talk-proposal.mdx`, `status.mdx`.
`startHere: ["status", "releases", "getting-started", "api"]` accounts for 4 of these 6.

### Orphans found: 2

1. **`docs/articles/contributing.md`** — builds to `/docs/contributing/` (confirmed present in
   `docs/build/docs/contributing/index.html`). Has `sidebar_position: 5` in its frontmatter, which
   is dead weight here — `startHere` is an explicit array, not autogenerated, so `sidebar_position`
   has no effect and the page doesn't appear in any sidebar. The `5` strongly suggests this page
   was meant to sit after `api` (position 4, implicitly) in the Start Here nav and was dropped when
   the array became explicit, rather than being deliberately orphaned.
2. **`docs/articles/sotm-2026-talk-proposal.mdx`** — builds to `/docs/sotm-2026-talk-proposal/`
   (confirmed present in the build output). This one **is** intentional: `sidebars.ts`'s own
   comment names it explicitly — _"these live at the articles/ root alongside orphans like
   sotm-2026-talk-proposal that should stay out of the nav."_

### Adjacent but out of scope for "orphan" (excluded from the build outright, not merely un-navved)

These are **not reachable by URL at all** — Docusaurus's `exclude` glob removes them from the docs
plugin's content before routes are generated, so they 404 (hard, not soft) rather than existing as
unlisted pages:

- `docs/articles/reviews/` — 15 files, entirely excluded (`reviews/**`).
- 38 `evals/**/*postmortem*` files + 1 `evals/**/*night-shift-session-report*` file — 39 files,
  all under `evals/night-shifts/` and `evals/retrospectives/`.

Total excluded-from-build: 54 files. (One of the 38 postmortems,
`evals/night-shifts/2026-06-18-night-postmortem.md`, also happens to be the one file in the repo
with `draft: true` frontmatter — belt-and-suspenders, since the glob exclude already removes it.)

## 4. Duplicate / near-duplicate titles

**Scope:** frontmatter `title:` field, swept across all of `docs/articles/` **excluding only**
`reviews/` (truly unpublished, not part of the site under any circumstance). Per the task, this
sweep **includes** `evals/` and `retrospectives/`, including the 39 postmortem/session-report files
that are excluded from the live build — flagged below wherever a match involves one of those, since
a title collision there can't actually manifest as a live search collision, only as a repo-hygiene
one.

**Result:** of 370 `.md`/`.mdx` files scanned, only **179 have a `title:` frontmatter field**;
**191 do not** (they rely on Docusaurus's fallback to the first H1 heading). This is itself a
finding: roughly half the corpus's page titles are undeclared and this sweep can't see them. The
191 no-title files cluster heavily in `evals/` (calibration, competitive-parity,
model-versions, night-shifts, resolver-geo) — dated report-style docs that were never given
explicit frontmatter titles.

### Exact duplicates: 1 pair

| Title            | Files                                                                                        |
| ---------------- | -------------------------------------------------------------------------------------------- |
| `Retrospectives` | `docs/articles/evals/retrospectives/index.mdx` and `docs/articles/retrospectives/README.mdx` |

Both are live (neither matches an exclude glob) — this is a genuine site-wide title collision: two
different sidebar sections (`Eval reports` and `Retrospectives`) each have a landing page titled
exactly "Retrospectives".

### Also a collision, but missed by a title-only sweep (worth flagging anyway)

- **`docs/articles/understanding/README.mdx`** has `title: Start here`. This is the index/landing
  page of the **`Understanding Mailwoman`** switcher section — but "Start here" is also the exact
  label of the **`startHere`** switcher section (a different, unrelated section containing
  Status/Releases/Getting started/API). A reader told to "start here" inside Understanding Mailwoman
  and a reader clicking the "Start here" section tab land on two different pages with the same name.

### Near-duplicates

1. **Likely true content duplication, not just naming** — `Record-matcher data catalog`
   (`docs/articles/concepts/record-matcher-data-catalog.md`) vs `Record-matcher source-data catalog`
   (`docs/articles/plan/reference/record-matcher-sources.md`). Read both: same subject (the public
   datasets the record-matcher joins), same data root path
   (`$MAILWOMAN_DATA_ROOT/record-matcher/sources/`), overlapping prose ("no shared key across
   publishers" / "no shared entity key across them"). One lives under Concepts, the other under
   Implementation Plan → reference, with no cross-link visible in the opening section of either.
2. **Same-day near-identical pair, both created today (2026-07-14)** —
   `Failure report — shipped · v257 · v261` (`evals/competitive-parity/2026-07-14-failure-report-golden.mdx`)
   vs `Failure report — shipped · v257 · v261 · v280`
   (`evals/competitive-parity/2026-07-14-failure-report.mdx`). Titles differ only by a trailing
   `· v280`. Worth a human check on whether the non-`-golden` one supersedes the other or they're
   both meant to coexist.
3. **`Postcode-only diagnostic (v0.6.0)`** (`evals/model-versions/2026-05-29-postcode-diagnostic.mdx`)
   vs **`Postcode-only geocoding`** (`understanding/alternatives/simple-postcode-only.mdx`) — share
   a prefix but are different documents in different sections (an eval report vs. a "simple
   alternatives" critique page); low actual confusion risk.
4. **Themed clusters** — same construction repeated across many titles, low individual confusion
   risk but visually noisy in any flat list/search-results view:
   - "Falsehoods about …" × 8 (addresses, format, shapes/dimensions, admin hierarchy, geocoded
     precision, numbers, postcodes, street names) — all `understanding/why-its-hard/falsehoods-*.mdx`,
     intentional parallel series.
   - `How it {used to work, will work, works now}` — 3 titles,
     `understanding/our-approach/how-it-*.mdx`, intentional parallel series, differ only by tense.
   - `Switching from {libpostal, Nominatim, OpenCage, Pelias, Photon}` — 5 titles,
     `concepts/switching-from-*.mdx`, intentional and well-differentiated (product names carry the
     distinction), lowest concern of the clusters.
   - `Night {shift, Shift, N postmortem}` naming — roughly a dozen titles under
     `evals/night-shifts/`, inconsistent even in capitalization (`Night shift 2026-06-18 — …` vs
     `Night Shift 2026-07-01 — Issue Cleanup`, capital vs lowercase "Shift"). Most of the raw
     postmortem files don't carry a `title:` at all (H1-only), so this list is smaller than the
     underlying file count suggests — but the inconsistent capitalization on the ones that do have
     a title is a real, fixable nit.

**Commands / method:** a small Python pass (not committed anywhere) that reads each file, extracts
the first YAML frontmatter block if present, pulls the `title:` line, strips surrounding quotes,
and groups by exact string match. Near-duplicates were then found by eye over the sorted unique-title
list (178 unique strings for 179 titled files) — this part is manual and could miss subtler
rewordings; treat the near-duplicate list as representative, not exhaustive.

## 5. Click paths for the four goal questions

Traced from the docs landing surface (homepage `/` and the top navbar) through the **current**
nav — DocsSubHeader switcher confirmed to be server-rendered (present in the static HTML, not a
client-only widget), so link targets below are exact, read directly out of
`docs/build/docs/status/index.html` and `docs/build/docs/plan/SCOPE/index.html`.

**Correction to the run-docs SKILL.md runbook:** its Gotchas section says _"The nav link labelled
'Docs' points there [`/docs/understanding/`]."_ That's stale. The built navbar currently emits
`<a href=/docs/status>Docs</a>` — the `startHere` sidebar's first doc, not `understanding`. This
looks like the SKILL doc predates the current `startHere` array (`["status", "releases",
"getting-started", "api"]`); worth a follow-up fix to that skill file separately from this baseline.

### (a) What does Mailwoman do and when should I use it?

Homepage (`/`) already answers this at zero clicks — the hero tagline, subtitle, and the
Parse/Geocode/Match "Gallery" cards are a solid capsule answer without any navigation. **The
problem is the navbar "Docs" link**: it goes to `/docs/status`, a dense, changelog-style page
("Verified as of 2026-07-02 — release 5.0.0", per-tag score callouts, a link to the parity
scorecard) — not an introduction. The actual "what and why" narrative
(`concepts/what-mailwoman-is.mdx`, `understanding/README.mdx`) requires clicking the DocsSubHeader
switcher to "Understanding Mailwoman" or "Concept deep dives" from there — a section switch a
first-time visitor has no particular reason to make from a status/changelog page. **Verdict: the
homepage nails it; the "Docs" nav entry point does not — adequate only if the visitor never leaves
the homepage.**

### (b) How do I parse, geocode, or integrate it?

Two good paths converge here: the homepage's "Read the docs" button goes directly to
`/docs/getting-started` (1 click, has install + first-parse code for both Node and CLI). The
navbar "Docs" → `/docs/status` → click "Getting started" in the `startHere` sidebar (2 clicks) gets
there too. Task-specific integration content (batch geocoding, multi-service geocoding, map
display, CSV ingest, timezone/UN-LOCODE lookups) lives in the `Recipes` switcher section, one more
switcher click away from any `startHere` page. **Verdict: adequate, best-served of the four — short
paths, code-forward, both entry points converge on it.**

### (c) How does the system work at a useful technical level?

The right content exists and is good (`concepts/staged-pipeline-contract.mdx`,
`understanding/our-approach/the-staged-pipeline.mdx`, the whole Concepts and Understanding tracks),
but nothing on the homepage or in the top navbar points at it directly — it's reachable only by
landing anywhere in `/docs/` and then using the DocsSubHeader switcher, whose existence isn't
signposted before you arrive on a docs page. From navbar "Docs": `/docs/status` → switch to
"Understanding Mailwoman" or "Concept deep dives" (2 clicks, plus discovering the switcher).
**Verdict: content is adequate once found; the path to it is not obvious from either of the two
site entry points (homepage, navbar).**

### (d) What is the authoritative contract or operational procedure?

This is the roughest path of the four. `plan/reference/SCHEMA.mdx` (the `ComponentTag` union
source of truth, per `AGENTS.md`) and `plan/reference/OPERATIONS.mdx` (the ops playbook) live
inside the **`Implementation plan 🧪`** switcher section — labelled with a test-tube emoji that
reads as "experimental / in-progress," which undersells content `AGENTS.md` calls authoritative.
Concretely, from navbar "Docs": `/docs/status` → switch to "Implementation plan 🧪" (lands on
`/docs/plan/SCOPE`, itself a good orientation doc, sidebar_position 1) → scroll the sidebar to find
a **collapsed, lowercase, auto-labeled** category literally named `reference` (no `_category_.json`
giving it a friendly label) → expand it → click into the specific doc. That's **3 clicks past the
navbar plus a manual expand**, and clicking the `reference` category header itself (rather than the
caret) navigates to `/docs/plan/reference/tokenizer-a0-baseline` — an unrelated tokenizer report,
not the schema or ops doc, since Docusaurus routes a category click to its first child by file
order. `SCHEMA.mdx` itself has **zero frontmatter** (no `title:`, no `sidebar_label:`), so its
sidebar entry is whatever Docusaurus auto-derives from the filename. **Verdict: inadequate — the
content that most needs to read as "authoritative" is the hardest of the four to reach, sits behind
a label that signals the opposite of authoritative, and its landing doc has no title of its own.**
