# Documentation architecture cleanup — implementation plan

**Status:** revised draft — review edits applied 2026-07-14
**Scope owner:** documentation maintainers
**Coordination boundary:** the evals and retrospectives inventory, curation, and migration are owned by a separate agent. This plan must not move, rewrite, reclassify, or change navigation for those sections without an explicit handoff.

## Goal

Make the public documentation easy to enter, trustworthy to cite, and inexpensive to keep current. A reader should be able to answer these questions without browsing dated implementation records:

1. What does Mailwoman do, and when should I use it?
2. How do I parse, geocode, or integrate it?
3. How does the system work at a useful technical level?
4. What is the authoritative contract or operational procedure?

This is an information-architecture and content-lifecycle change, not a project to rewrite every document or erase the project's technical history.

## Audience and vocabulary

The core reader is a cartography/geodata practitioner, not an ML practitioner. Legacy parsers are intuitive for a reason: Pelias and libpostal are rules you can read and gazetteers you can grep, and a reader can predict their behavior from the docs alone. An encoder is simple, mature machinery by modern LLM standards, but its vocabulary is statistical — embeddings, emissions, priors, calibration — and these docs sit exactly at the marriage of those two worlds. That junction is where we lose the audience.

Two rules follow:

- Introduce every learned mechanism through its rule-world analog before naming the math: the gazetteer lookup the reader already trusts becomes an FST prior; a hand-written pattern becomes a learned per-token score; tie-breaking heuristics become the Viterbi best path. Analog first, term second, equation (if ever) last.
- No page in Start here, Use Mailwoman, or the entry layer of Understand may require ML background to answer its product question. Statistical depth lives one link away, never in the entry path.

## Evidence and editorial model

The `articles/` tree holds ~385 pages: `evals/` 190 (delegated), `plan/` 60, `concepts/` 50, `understanding/` 46, `reviews/` 15 (already excluded from the build), `recipes/` 8, `licensing/` 6, `retrospectives/` 4, plus the root start-here pages. A separate `research/` blog holds ~44 dated posts. Every sidebar except the curated `startHere` list is an autogenerated directory dump. Automatic navigation is useful for a small homogeneous collection; it makes active guidance, implementation history, and evidence compete as peers at this scale.

Use the following books as process and pedagogical guides. They inform the structure and quality bar; no book text, diagrams, or examples are copied into the site.

- _Docs for Developers_ (Bhatti et al.): audience-first planning, code samples, navigation, feedback, measurement, and maintenance/deprecation.
- _Natural Language Processing with Transformers, Revised Edition_ (Tunstall, von Werra, Wolf): explain the parsing model from representation through token classification and decoding, before implementation-specific detail.
- _Designing Machine Learning Systems_ (Chip Huyen): distinguish the product/runtime path from the data, training, evaluation, and monitoring lifecycle.
- _Machine Learning Design Patterns_ (Lakshmanan, Robinson, Munn): make data provenance, reproducibility, evaluation, and constraints explicit rather than implicit folklore.

## Content model

Every maintained public page must declare its role in frontmatter and obey its contract.

| Role        | Reader need                           | Time horizon            | Canonicality              | Required fields                          |
| ----------- | ------------------------------------- | ----------------------- | ------------------------- | ---------------------------------------- |
| `guide`     | accomplish a task                     | evergreen               | one preferred path        | audience, prerequisites, verified-with   |
| `tutorial`  | learn by doing                        | evergreen but versioned | one learning journey      | audience, prerequisites, verified-with   |
| `concept`   | understand a stable idea              | evergreen               | one explanatory home      | audience, source-of-truth, review-by     |
| `reference` | look up a precise contract            | release-bound           | authoritative             | source-of-truth, generated-from or owner |
| `decision`  | understand an active choice           | active until superseded | links to accepted outcome | status, owner, superseded-by when closed |
| `evidence`  | inspect a result or historical record | immutable/datestamped   | cited by maintained pages | date, status, promoted-conclusions       |

Do not require a mechanical rename of every existing file. Add the fields as pages are touched, and enforce them first for new or substantially rewritten maintained pages.

## Target navigation

The top-level switcher should lead with reader intent. Its final labels are a design choice, but its roles are not:

```text
Start here       fork page · status · installation · first parse · API overview
Use Mailwoman    parse · geocode · browser · batch · privacy recipes
Understand       system overview · parsing model · resolution · data · quality
Reference        schemas · API contracts · architecture · runtime flags · data sources
Contribute       development · model work · operations · releases
Archive          dated plans · evidence · research/retrospectives (where deliberately public)
Legal            licensing · provenance · privacy · SBOM
```

`Start here`, `Use Mailwoman`, `Understand`, and `Reference` are curated lists—not autogenerated directory dumps. `Archive` is searchable, permalink-stable, and visibly dated, but does not appear as the default path for a new reader.

`Start here` opens with a fork-in-the-road page, not a package list. The current entry (`getting-started.mdx`) leads with a three-package `npm install` before the reader knows what Mailwoman is; the fork routes by intent instead:

- **"I want to parse addresses"** → installation, first parse, then Use Mailwoman.
- **"I want to understand how this works"** → the flagship parsing explainer, then Understand.
- **"I want to understand the problem"** (why postal addresses are hard) → the narrative track (`the-problem`, `why-its-hard`).

Package names appear after the fork, in the door the reader chose — never as the greeting.

Two existing trees the navigation must account for explicitly:

- `understanding/` (46 pages: `the-problem`, `why-its-hard`, `alternatives`, `our-approach`, `exotic-poi`) is the narrative track and the second-largest maintained section. It feeds `Understand` and the compatibility/migration material; each subtree gets an inventory disposition rather than a wholesale move.
- `research/` (~44 dated posts) is a blog and stays a blog: dated, permalinked, linked from maintained pages as evidence. It is not migrated into the docs tree.

## First flagship rewrite: encoder-based address parsing

Use the proposed “Encoder-Based Address Parsing: From Tokens to Postal Grammar” article as the starting point for **How Mailwoman parses an address**. It should be Mailwoman-specific, technically accurate, and short enough to orient a product reader before linking to deeper references.

Required outline:

1. Address parsing as sequence labelling, using Mailwoman's component names and one real input/output example.
2. SentencePiece/subword tokenization and the token-to-source-span distinction.
3. The contextual transformer encoder and per-token emissions.
4. BIO spans and the canonical `ComponentTag` schema (link; do not duplicate the union).
5. FST/gazetteer priors and the boundary between learned and deterministic signals.
6. Viterbi/BIO validity, tree construction, then resolver handoff.
7. Locale scope, failure modes, and links to current evaluation evidence.

Each mechanism enters through its rule-parser analog before its statistical name, per the audience rules above: gazetteer lookup → FST prior, hand-written pattern → learned per-token emission, tie-breaking heuristics → Viterbi best path. A Pelias-literate reader should reach the end with their existing intuitions extended, not replaced.

Keep BiLSTMs as a brief historical comparison, if at all. Do not claim a shared multilingual model, learned CRF transitions, a model geometry, supported locales, or a data source without verifying it against the current implementation and the source-of-truth reference. Avoid generic title-case labels such as `HouseNumber` when the public contract uses a different spelling.

## Work plan

### Phase 0 — baseline and guardrails

- [ ] Record a content inventory outside the evals/retrospectives scope: path, current sidebar, title, last substantive change, role, intended audience, owner, status, inbound links, and recommended action (`keep`, `rewrite`, `merge`, `archive`, `remove nav`).
- [ ] Identify the canonical source for each reference claim: code-derived contract, schema, generated artifact, or named owning document.
- [ ] Give `understanding/` (each of its five subtrees) and the `research/` blog explicit dispositions in the inventory.
- [ ] Flag known stale claims for the inventory: `concepts/crf-decoder.mdx` (the CRF has been CE-only since v0.5.0 — no learned transitions ship) and any page implying a shared multilingual model (fr-fr ships the en-us weights).
- [ ] Capture baseline measures: number of primary-nav entries, broken links, orphan pages, duplicate/near-duplicate titles, and the click paths for the four goal questions.
- [ ] Agree an archive/publicness policy before exposing or moving internal material. Existing Docusaurus exclusions remain untouched by this plan.

**Exit criteria:** reviewed inventory for all non-eval/non-retrospective pages; no content moves yet.

### Phase 1 — navigation curation

The navigation machinery already exists: the section switcher (`src/components/DocsSubHeader`) with one sidebar per section, a curated `startHere` list, and DocSearch. This phase re-labels and curates what's there; it builds nothing new. Sidebar ids in `sidebars.ts` and the section list in `src/components/DocsSubHeader/sections.ts` are coupled — they must list the same ids in display order and move together in the same PR.

- [ ] Curate the sidebars and section-switcher labels using the target navigation above, replacing autogenerated dumps for the reader-facing sections.
- [ ] Decide the permanent home and visual treatment for active decisions versus immutable historical records.
- [ ] Add a compact “documentation map” page that tells readers which section is authoritative for guides, reference, active work, and archived evidence.

**Exit criteria:** navigation PR is reviewable without content deletion; moved pages remain findable through search and the documentation map. Old inbound links are deliberately not preserved — no redirect infrastructure is added.

### Phase 2 — durable reader paths

- [ ] Rewrite Start here around the fork-in-the-road page and a five-minute first parse with clear runtime/environment prerequisites. Reconcile the entry-point contradiction while there: `getting-started.mdx` teaches `NeuralAddressClassifier` directly, but the API reference names `createRuntimePipeline` as the recommended entry point — the first parse a reader runs should use the recommended path.
- [ ] Curate Use Mailwoman from the existing recipes and API material; every guide has a stated outcome and a verified code example.
- [ ] Publish the flagship parsing explainer, then establish the other canonical concept pages one at a time.
- [ ] Replace duplicated prose with short transition pages or contextual links only after the canonical page is accepted.
- [ ] Ensure each maintained concept links to its relevant reference and to evidence without embedding dated experiment narration.

**Exit criteria:** the four goal questions have one primary answer each; prominent concepts have an explicit owner and a source-of-truth link.

### Phase 3 — reference and contributor boundaries

- [ ] Separate generated/API contracts from explanatory prose. Generated data is generated during the docs build or linked to its generation source; it is not manually copied.
- [ ] Make `SCHEMA`, interfaces, architecture, runtime flags, data sources, operations, and release/runbook material discoverable from Reference or Contribute according to audience.
- [ ] Mark active design records with status and successor links; do not present historical plans as current roadmap.
- [ ] Add an explicit “current scope and roadmap” entry point that points to the maintained scope declaration rather than the plan directory.

**Exit criteria:** a reader can distinguish an API/schema contract, an active decision, and a historical plan from page chrome and first screen.

### Phase 4 — prevent relapse

- [ ] Add a contributor-facing documentation policy: page roles, required frontmatter, ownership, when to write a new page versus update a canonical one, and how to retire a page.
- [ ] Add CI checks proportionate to the site: link/build validation, frontmatter validation for maintained roles, duplicate title detection, and an allowlisted orphan-page report.
- [ ] Add review prompts to documentation-affecting pull requests: reader question, canonical page updated, factual source, executable/verified examples, redirects, and review date.
- [ ] Schedule a quarterly lightweight “freshness sweep” for pages marked `review-by`; this is not a requirement to churn archival evidence.

**Exit criteria:** contributors have a clear path for adding, updating, and retiring documentation; basic structural regressions fail before merge.

## Non-goals

- Rewriting every historical plan, eval report, or research post into evergreen prose.
- Deleting dated evidence solely because it is old.
- Changing product behavior, component names, schemas, or model architecture to suit documentation.
- Introducing a CMS or a large documentation platform migration.
- Overlapping the delegated evals and retrospectives cleanup.
- Preserving old inbound links: no redirect infrastructure. Moved pages stay findable through search and the documentation map.
- Migrating the `research/` blog into the docs tree.

## Acceptance measures

Measure the change after the migration rather than declaring success from file count alone:

- A first-time developer completes the first parse path without entering `plan/` or an archived page.
- Each primary reader question has a single canonical page reachable from the top-level docs navigation in at most two section choices.
- All maintained public pages have a declared role and owner/source-of-truth; dated evidence is visibly dated and contextualized.
- Docs build and link checks pass, and navigation contains no accidental autogenerated archival dump.

## Handoffs and sequencing

1. Review and accept the content model and target navigation.
2. Hand this plan's role taxonomy (`evidence`, `decision`, required fields) to the evals/retrospectives agent before they finalize theirs, then receive their taxonomy and publicness recommendations expressed in the shared vocabulary — one frontmatter vocabulary across the site, not two.
3. Land Phase 1 navigation/redirect infrastructure.
4. Land flagship reader paths in small, independently reviewable PRs.
5. Enable lifecycle checks after the canonical homes exist, so the checks guide future work rather than obstruct the cleanup.

No evals or retrospectives change is included in this plan until step 2 completes.
