## What shipped this session (verified live)

### 1. v4.15.0 — v1.9.3a3 anchor-absorption (the #723 fix) — PRs #796, release `cf01e991`

The postcode anchor was firing on a leading 5-digit token even when it was a US house
number that happens to be a real ZIP (`12345 Main St`). Root cause: a **train/inference
WHERE-mismatch** — training painted the anchor on _gold_ postcode spans, inference paints
on postcode-_shaped_ spans, so the model never trained on the failing case. Fix =
`anchor_paint_mode: shaped` (load-time transform, no corpus rebuild) + a
`synth-anchor-absorption` counter-shard with a CASE-H/CASE-P **locality discriminator**
(converged via an A0→A3 probe ladder).

- **Result**: SLICE-H (leading-5-digit house#, anchor ON) **20→100**, at **zero coordinate
  cost** — `oa-resolver-eval` (2000 OA-US, full ship-config): whole-US locality 97.7→97.8,
  p50/p90 flat, p99 275→259; VT (the E911 leading-postcode state where the trade lives)
  FLAT 92.6.
- **Cost**: `us.postcode` label-F1 −2 (95.0) + `fr.postcode` −0.2 (99.3) — both
  coordinate-invisible.
- **Gate revised** separately (no silent drift): `scripts/eval/gates/v4.15.0-boundary.json`
  lowers only those two postcode floors with the coordinate justification stated. Re-graded
  **PASS** (every floor, mask-regression PASS, int8↔fp32 ≤0.8).
- **int8 md5** `4dec4f460a934949580d8e7b43adae7e` — consistent across /mnt/playpen → HF → R2
  → npm pack.
- Doc: `docs/articles/evals/2026-06-25-v4.15.0-promotion.md`.
