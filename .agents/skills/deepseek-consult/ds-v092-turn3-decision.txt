**Q1 – Why dual‑injection did nothing**

The position‑0 injection was redundant because the model already has full self‑attention over the sequence, including the per‑token anchor at the postcode. The harm stems not from lack of availability but from the **learned functional directionality** of the anchor vector itself: it acts as a *rightward‑seeking locality signal* – it biases attention from the postcode toward tokens to its right. In native order that targets the city; in international order it targets the region (or nothing), while suppressing the actual city on the left. Adding the same vector at position 0 merely adds a second, equally‑directional signal that the model either ignores or tries to use but fails to re‑route effectively. The vector’s association with “locality follows the postcode” is baked into the training. The model *cannot* spontaneously reinterpret an identical vector at a different location as “locality precedes the postcode” without targeted training, and a single additive injection does not provide the necessary disambiguation.

**Q2 – Decision: Accept the asymmetry. Ship the native win.** **(b)**

Justification:
- Three independent retrains failed to close the gap; the native boost (+35 pp) is robust, large, and generalises (US, FR hold). The international penalty (−4 pp) is small relative to the gain and is *stable* across all attempted fixes.
- Anchor‑off international already hits ~48% locality‑match, which is competitive with non‑neural baselines. The model can serve international‑order DE via `c=0` (identity, just the base parser) with a simple per‑request toggle based on a lightweight order heuristic (e.g., postcode position relative to locality in the raw string – trivial to implement with no extra model cost). No inference‑time anchor on international addresses.
- The anchor approach is **not killed**; it is provably order‑asymmetric but this is a local phenomenology we now understand. For production, per‑locale models can train anchors for the dominant order; for a universal model, we treat international orders as an explicit fallback path. The anchor’s value for native orders across all future locales is too high to abandon. I prioritise the product win over a speculative architectural fix that would consume the remaining budget and likely fail for structural reasons.

**Q3 – If the anchor asymmetry is cross‑cutting, does it kill the multi‑locale neural parser plan? What’s the highest‑value next thread instead?**

It does **not** kill it; it constrains it. The anchor will always strongly prefer the order in which the postcode precedes the locality. To serve both orders in a single model without a fallback, we would need to make the anchor’s locality‑binding *bidirectional*, which requires training the model to associate the anchor with “locality token, regardless of whether it appears to the left or right.” That is a learnable behaviour but requires explicit mixed‑order training with disambiguating signals – something we haven’t yet tried because the current training only mixes orders (no explicit loss for anchor misbinding). That is a legitimate follow‑up research thread, but not a one‑run fix.

The **highest‑value next thread**: move the anchor to **locales where the native postcode order matches the typical US/international order** – i.e., postcode‑then‑city. Many countries (US, FR, GB, NL, etc.) already place postcode after city? Wait, check: US: “Seattle WA 98101” postcode after city, same order as international (city then postcode). So the anchor, injected at the postcode, would be *trailing* the city in those cases. That conflicts with the German native success. Actually German native order is postcode‑before‑city, which is atypical. For US/FR, native order is “street house#, city, state postcode” – postcode after city. In that case, anchor injected at the postcode would trail the city, which is the exact harmful configuration we saw for German international. So the anchor might *hurt* US locality too, unless the anchor is trained to work in that configuration. But the results say US 96.4 / FR 84.9 held, meaning the anchor didn't hurt US. That's interesting: US anchor-on is 96.4% (pilot similar), anchor-off presumably similar? They didn't report anchor-off US, but if it's similar, that means the anchor doesn't harm US. Why? Because the US anchor is trained with US addresses where postcode follows city, so the anchor learned to associate “locality precedes the postcode” (leftward). So the directionality is learned from training data. In US training, postcode always came after city; thus the anchor vector learned to look left. In German training, the native order has postcode before city, teaching “locality follows postcode” (rightward). The mixed‑order German training taught both orders, but the anchor vector is a single learned parameter; it cannot simultaneously optimise for both directions if they conflict. The anchor likely converged to an average that hurts the trailing case because the native order examples dominated (~60%). So the asymmetry is locale‑specific training, not an inherent structural flaw. For US, the anchor works because training data is consistent. So the multi‑locale plan can succeed by ensuring that the anchor’s direction matches the dominant order for each locale, either by separate anchor vectors per locale (feasible with a country‑conditioned anchor) or by training a single vector on globally dominant order (postcode‑after‑city) and handling the exceptional locales (DE, NL, etc.) separately. That is the highest‑value thread: split the anchor by country or use a country‑conditioned vector, which would allow the DE anchor to specialise on native order and the US/anchor on international‑style order, without conflict.
